# 🗄️ HiTsBOT - Database Architecture & Schema Specifications

This document defines the persistent data layers, cache strategies, and Object-Relational Mapping (ORM) rules for the **HiTsBOT** infrastructure. It bridges the Gap between the Core API and the WhatsApp Event Worker.

---

## 1. Storage Strategy Overview

HiTsBOT utilizes a hybrid architecture split across three distinct tiers to optimize performance, manage state resilience, and maintain token budget efficiency.

| Storage Tier | Technology | Technical Purpose / Domain Scope |
| :--- | :--- | :--- |
| **Relational Database** | MySQL (v8.0+) | Persistent store for mission-critical application configurations, user encryption hashes, API keys, and device metadata. |
| **In-Memory Cache** | Redis (v7.0+) | Sub-millisecond engine powering the outbound messaging queue (BullMQ) and managing live user context memory threads (last 5 chat logs). |
| **Local File System** | Node.js File System | Persistent session multi-auth state storage (`useMultiFileAuthState`) generated by Baileys. Safeguards connection tokens without overloading MySQL with binary/large string writes. |

## 2. Prisma Schema Specification (MySQL Dialect)

The Prisma schema must be initialized inside `apps/backend-core/prisma/schema.prisma`. It utilizes optimized field types for high-density string payloads (such as AI guidelines).

```prisma
datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

/// Core administrator account model for Dashboard access
model Admin {
  id        Int      @id @default(autoincrement())
  username  String   @unique
  password  String   // Enforced bcrypt hash storage
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([username])
}

/// Core configuration state for the WhatsApp Bot and integrated AI Model
model BotConfig {
  id                Int      @id @default(autoincrement())
  botName           String   @default("HiTsBOT Agent") @db.VarChar(100)
  waNumber          String?  @unique @db.VarChar(20)
  waStatus          String   @default("DISCONNECTED") // Reference tokens: CONNECTED, DISCONNECTED, INITIALIZING, SCANNING
  aiProvider        String   @default("gemini") // Supports: gemini, openai
  aiApiKey          String?  @db.VarChar(255)
  systemInstruction String?  @db.Text // Maps directly to MySQL LONGTEXT/TEXT data types to support massive prompt layouts
  aiTemperature     Float    @default(0.1) // Hard-capped at 0.1 - 0.2 inside business logic layers to minimize hallucination
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  @@index([waNumber])
}
3. Redis In-Memory Context Layout
To ensure the AI response remains conversational without exhausting token quotas, the backend intercepts events and maintains a rolling chat log queue natively inside Redis hashes or sorted sets.

Context Retention Rules:
Maximum Threshold: Exactly 5 messages per unique customer phone number (JID).

Data Lifecycle: Set an explicit TTL (Time to Live) of 24 Hours on every customer conversation key. If a customer is inactive for a day, the context safely purges to save Redis cluster memory.

Key Structure Scheme: hitsbot:context:{whatsappNumber}:{customerJid}

Key Representation Example:
JSON
// Redis Key: hitsbot:context:628123456789:6289999999@s.whatsapp.net
[
  {"role": "user", "parts": "Hi, do you sell red shoes?"},
  {"role": "model", "parts": "Yes! We have red shoes available in sizes 40 to 44."},
  {"role": "user", "parts": "Are they currently in stock?"}
]
4. BullMQ / Message Broker Blueprint
All outbound API transactions generated by the AI model are queued within Redis via BullMQ before hitting the Baileys network adapter.

Queue Namespace: hitsbot-outbound-messages

Job Payload Constraints:

TypeScript
  interface OutboundJobPayload {
    toJid: string;        // Target recipient customer JID
    messageText: string;  // Cleansed AI generation response string
    attemptCount: number; // Retry tracking metrics (Max retries: 3)
  }
Job Options Strategy: Jobs must trigger back-off delays upon failures (attempts: 3, backoff: { type: 'exponential', delay: 1000 }) to prevent hitting WhatsApp network rate-limits.

5. Performance & Safety Checklist for AI Models
Prisma Type Enforcements: Ensure systemInstruction is strictly annotated with @db.Text or @db.LongText. Storing this inside a standard MySQL VarChar(255) string wrapper will instantly truncate your AI prompts and cause application breaks.

Connection Pools: Configure MySQL connection pooling inside your deployment environment configurations to balance requests from the API cluster and asynchronous message workers efficiently.
